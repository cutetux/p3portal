# OpenTofu Engine Foundation (P3 Stacks – Phase 2a)

> **Edition:** P3 Plus only. The Core image ships no OpenTofu binary and no provider
> mirror (`tools/verify-core-build.sh` enforces this).
>
> **Scope of this document:** the *engine foundation* — binary bundling, the provider
> offline mirror, the per-node `tofu` token, local-on-volume state with native encryption,
> and a manual lifecycle proof (`apply` / `plan` / `destroy`) against a real Proxmox host.
> There is **no** P3 deploy flow in Phase 2a (no `/api/stacks/.../deploy`, no transpile,
> no job). That arrives in Phase 2b.

## Why OpenTofu, and what about "other" VMs?

OpenTofu only ever touches resources recorded in **its own per-stack state file**. VMs and
LXC containers created through the Proxmox UI, Ansible, or another stack are **not** in that
state, so `plan` / `apply` / `destroy` / drift never see or change them. The state file is
the hard isolation boundary and the technical implementation of the "one engine per resource"
rule. There is no `import` and no node-wide data source — Proxmox assigns the VMID, so there
is no collision with foreign VMs.

## Image bundling & the offline provider mirror

The Plus build (`EDITION` ≠ `core`) does the following in the Dockerfile:

1. Downloads the pinned OpenTofu binary, verifies it against the official
   `tofu_<ver>_SHA256SUMS`, installs it to `/usr/local/bin/tofu`.
2. Runs `tofu providers mirror -platform=linux_amd64 /opt/tofu/plugin-mirror` against a
   throwaway `required_providers` file — this needs network **at build time only** (exactly
   like the packer plugin install).
3. Writes `/opt/tofu/tofurc` with a `filesystem_mirror` block and **no `direct` block**, and
   sets `ENV TF_CLI_CONFIG_FILE=/opt/tofu/tofurc` globally.

Result: at runtime `tofu init` resolves the provider purely from
`/opt/tofu/plugin-mirror/registry.opentofu.org/bpg/proxmox/<ver>/linux_amd64/` — **no call to
`registry.opentofu.org`**. The mirror lives in the image (`/opt/`), not in the data volume.

```bash
# In the running Plus container:
tofu version                       # → prints the pinned version
ls /opt/tofu/plugin-mirror/registry.opentofu.org/bpg/proxmox/   # → <version> dir present
```

## The `tofu` token (5th per-node tier)

The OpenTofu engine authenticates with a dedicated, optional, per-node token (`PortalTofu`
role — see `docs/proxmox-setup.md` Part D). It is stored Fernet-encrypted in the `nodes`
table (columns `tofu_token_id` / `tofu_token_secret`) like the packer token and is **never**
written into HCL or the state plaintext.

At runtime the engine injects the bpg-provider auth as **environment variables** (Tech-Design
Open Point 4):

| Env var | Value |
|---|---|
| `PROXMOX_VE_ENDPOINT` | `<node.url>` (base URL, no `/api2/json`) |
| `PROXMOX_VE_API_TOKEN` | `<tofu_token_id>=<tofu_token_secret>` |
| `PROXMOX_VE_INSECURE` | `true` when the node has `verify_ssl` disabled |

The provider block in the `.tf` therefore carries **no inline credentials**.

## State on volume + native encryption

- **Backend:** local, one state file per stack at
  `/app/data/stacks/<stack_id>/terraform.tfstate`. Volume-persistent → survives container
  restart. The working directory equals the state directory.
- **Encryption:** OpenTofu native state encryption (≥ 1.7), `pbkdf2` key provider +
  `aes_gcm` method on both `state` and `plan`. The passphrase comes from
  `/app/data/tofu_state.key` (32-byte hex, `chmod 600`, generated by the entrypoint if
  missing) and is injected via the **`TF_ENCRYPTION`** environment variable — never written
  into a `.tf` file.

**Why a separate key file and not `SECRET_KEY`?** `SECRET_KEY` is deliberately rotatable
(it signs JWTs and powers Fernet encryption). If the state passphrase were derived from it, rotating `SECRET_KEY`
would irreversibly break every encrypted state. The dedicated `tofu_state.key` decouples the
two. It lives in the same data volume as the state, so a single backup ("back up
`/app/data/`") covers both.

> ⚠️ **Backup duty:** losing `tofu_state.key` makes every encrypted stack state
> **unrecoverable**. Back up `/app/data/` (key + state together).

## Concurrency lock

A module-global `dict[str, asyncio.Lock]` (`backend/plus/stacks/engine.py`, `get_stack_lock`)
serialises tofu runs per stack — "one apply per stack". In Phase 2a this is proven only
mechanically (unit test); the HTTP-409 wiring arrives with the deploy job in Phase 2b.

## Lifecycle proof (run manually against a real Proxmox)

This proves the plumbing end-to-end. **CI does not run this** (no Proxmox cluster in CI; CI
only checks that the binary + mirror are present in the Plus image and absent in Core). Use
the reference `.tf` in `docs/examples/opentofu/` (clearly marked as a reference, not app
code).

### 0. Prerequisites

- A Plus container with a configured `tofu` token on the target node.
- An existing VM **template** on the node (e.g. a cloud-init Debian template).
- `tofu_state.key` present in `/app/data/` (auto-generated on first Plus start).

### 1. Prepare the working directory and env

```bash
# Inside the Plus container
STACK_ID=proof-01
WORKDIR=/app/data/stacks/$STACK_ID
mkdir -p "$WORKDIR"
cp /app/docs/examples/opentofu/clone-vm.tf "$WORKDIR/main.tf"
cd "$WORKDIR"

# Per-node bpg auth (replace with your node's values)
export PROXMOX_VE_ENDPOINT="https://pve01.example.com:8006"
export PROXMOX_VE_API_TOKEN="portal-tofu@pve!portal-tofu=<SECRET>"
export PROXMOX_VE_INSECURE="true"          # if the node uses a self-signed cert

# Native state encryption (passphrase from the key file, never in a .tf)
export TF_ENCRYPTION="$(cat <<EOF
key_provider "pbkdf2" "p3" { passphrase = "$(cat /app/data/tofu_state.key)" }
method "aes_gcm" "p3" { keys = key_provider.pbkdf2.p3 }
state { method = method.aes_gcm.p3 }
plan  { method = method.aes_gcm.p3 }
EOF
)"
```

> In the product this env is built by `backend/plus/stacks/engine.py`
> (`build_tofu_env(node)`); the snippet above mirrors it for the manual proof.

### 2. `init` (from the offline mirror) + `apply`  — AC-2A-LIFE-1

```bash
tofu init      # resolves bpg/proxmox from /opt/tofu/plugin-mirror, no network
tofu apply -auto-approve
```

Expected: a new VM appears in Proxmox, cloned from the template.

### 3. Verify the encrypted state  — AC-2A-STATE-2

```bash
head -c 200 terraform.tfstate
# Expected: an encrypted envelope (NOT readable plaintext JSON), e.g.
# {"encryption_version":"v0","crypto":"...base64..."}
```

### 4. Out-of-band change → drift  — AC-2A-LIFE-2

Change the VM's CPU count in the Proxmox web UI, then:

```bash
tofu plan -json | tofu show -json - 2>/dev/null || tofu plan
# Expected: the plan reports the VM as "update" (changed) — proves drift detection,
# which Phase 2b surfaces in the UI.
```

### 5. `destroy`  — AC-2A-LIFE-3

```bash
tofu destroy -auto-approve
# Expected: the VM is removed from Proxmox; the state no longer tracks resources.
```

### 6. Record the evidence  — AC-2A-LIFE-4

Paste the relevant `apply` / `plan` / `destroy` excerpts below when you run the proof in your
environment:

```
# tofu apply  → "Apply complete! Resources: 1 added, 0 changed, 0 destroyed."
# tofu plan   → "Plan: 0 to add, 1 to change, 0 to destroy."  (after the CPU change)
# tofu destroy→ "Destroy complete! Resources: 1 destroyed."
```

## What is intentionally *not* here (Phase 2a guards)

No `/api/stacks/{id}/deploy` endpoint · no P3-model → HCL transpile · no job-system / live-log
wiring · no mutation-block UI for stack VMs · no network/SDN/storage resources. The
`engine.py` helper is a plumbing module **not bound to any route**. All of this is Phase 2b.
