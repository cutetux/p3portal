# v1.74.8-beta — Clean-room re-implementation of Issue #3 / #4 security fixes

This release re-implements the security fixes for
[#3](https://github.com/P3Portal-org/p3portal/issues/3) (SQLite
`database is locked` under concurrent writers) and
[#4](https://github.com/P3Portal-org/p3portal/issues/4) (Zip-Slip via
absolute paths in `save_playbook_zip` / `save_template_zip`) using an
**independent implementation that does not derive from the code
suggested in the issue reports**.

## Why this release exists

GitHub Issues are not automatically covered by the repository's
licence. GitHub Terms of Service § D.5 grants only display rights on
the GitHub platform — not redistribution rights inside this codebase.
Previous releases v1.74.5-beta through v1.74.7-beta contained code
patterns adopted directly from the "Suggested Fix" blocks in the
issue reports, which leaves the contribution-licensing question
unresolved.

To remove that ambiguity, the affected functions have been rewritten
from scratch using a different design.

## What changed

- **`backend/services/_zip_safety.py`** (new)
  Central helper `extract_zip_safely()` used by both playbook and
  packer upload paths. Validation uses `pathlib.PurePosixPath`,
  containment check uses `os.path.commonpath()`, extraction uses
  chunked stream-copy via `zipfile.ZipFile.open()` in 64 KiB chunks.
  Custom `UnsafeZipMember(ValueError)` exception. Also rejects
  Windows drive-letter prefixes (`C:\foo`, `D:/bar`) that
  `PurePosixPath` leaves intact.
- **`backend/services/playbook_service.py`**,
  **`backend/services/packer_service.py`** — inline extraction loops
  removed, both call `extract_zip_safely()` and clean up the partial
  destination directory on failure.
- **`backend/db/database.py`** — `_set_sqlite_pragma()` now iterates
  a module-level `_SQLITE_CONNECTION_PRAGMAS` tuple and uses
  `try/finally` for cursor cleanup, instead of three inline
  `cursor.execute()` calls.
- **Tests** — `test_playbook_service.py` and `test_packer_service.py`
  deleted (they were derived from the issue's "Test to Add" block).
  Replaced by `test__zip_safety.py` with an independent test
  structure (parametrized class-based tests, different fixtures,
  different data, different assertions).
- **`backend/core/deps.py`** — unchanged from v1.74.7-beta. The
  previous "fix" there was deleting four lines that the bug report
  identified as a dead-code privilege escalation. No creative content
  was ever adopted from the report for that file.

## Functional behaviour is identical

Same protections, same external API, same exception semantics
(`UnsafeZipMember` is a subclass of `ValueError`, so existing
`except ValueError` callers keep working unchanged).

## Action for operators

**Upgrade from v1.74.5-beta / v1.74.6-beta / v1.74.7-beta to
v1.74.8-beta.** Functional behaviour is unchanged; the change is
licensing-hygiene only. The Docker images
`ghcr.io/p3portal-org/p3portal:latest`,
`ghcr.io/p3portal-org/p3portal:core`, and
`ghcr.io/p3portal-org/p3portal:plus` will point at the v1.74.8-beta
build automatically once CI completes.

### Plus-edition images for v1.74.5-beta..v1.74.7-beta removed

To stop continued proprietary distribution of source code with
unresolved contribution licensing, the Plus-edition images for the
superseded versions have been removed from the registry:

- `ghcr.io/p3portal-org/p3portal:1.74.5-beta-plus`
- `ghcr.io/p3portal-org/p3portal:1.74.6-beta-plus`
- `ghcr.io/p3portal-org/p3portal:1.74.7-beta-plus`

Operators who pinned a Plus image to any of these tags must move to
`:1.74.8-beta-plus` or `:plus`. Functional behaviour is unchanged.

The corresponding **Core-edition** images
(`:1.74.5-beta`, `:1.74.6-beta`, `:1.74.7-beta`) remain available
for reproducibility — Core is distributed under AGPLv3, under which
the contribution-licensing question is materially weaker than for
the proprietary Plus bundle. See [`SECURITY.md`](../../SECURITY.md)
for the policy detail.
